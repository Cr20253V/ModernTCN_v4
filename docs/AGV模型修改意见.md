# AGV 模型修改意见与分析

## 1. Mamba 算法核心目标
Mamba 算法的核心目标是**学习真实的 AGV 系统行为**，包括其嵌入式控制器的响应特性。
- **输入**：Simulink 模型的观测输出（sensor measurements）
- **期望输出**：AGV 的当前行驶状态（如 straight, turning, slope, slipping, stall）
- **关键约束**：Mamba 在部署时只能使用真实的传感器数据（IMU, 编码器电流/转速），不能使用仿真器内部的 "God-view" 状态（如真实的 `beta`, `mu`）。

## 2. 现有输出变量分析 (Output Equation)
分析文件 `src/core/output_eq_ref.m`，当前输出维度为 31。

### 2.2 缺失的关键信息
1. **滑移程度 (Slipping Degree)**: 目前只有 binary 的 `slip_flag`。Mamba 需要连续的滑移率信号来判断 "打滑程度"。
2. **横向加速度 (Lateral Accel)**: 只有 `v*omega` 的运动学估计，缺乏基于受力的加速度计读数（即使是非完整的）。
3. **俯仰动态**: `gyro_y` 缺失，导致无法感知坡度变化的瞬间。

## 3. 参数合理性审查 (Parameters)
分析文件 `src/core/parameters.m`。

### 3.1 关键物理参数
- `mass = 200 kg`: 合理。
- **`cornering_stiffness = 300 N/rad`**: **极度不合理**。对于 200kg 的车，载荷 ~500N/轮。300 的侧偏刚度意味着 1度侧偏就产生 5N 的力，这太软了，像气球。通常应在 3000-10000 级别。
    - *推测*: 之前可能为了掩盖模型的不稳定性而调低了。
    - **建议**: **暂时保留**，因为 Mamba 要学的是 *当前仿真模型* 的行为。但需记录此为 "Sim2Real Gap" 的风险点。

### 3.2 人工阻尼 (Artifacts)
`state_eq_ref.m` 中包含人为的阻尼项：
- `C_damping = 250.0` (横摩阻尼)
- `-5.0 * beta` (侧偏阻尼)
**结论**: 这些是 "God-mode" 的稳定控制。Mamba 将学习到一个 "自带稳定器" 的车辆动力学。这符合 "学习系统行为" 的目标。

## 4. 载荷变化 (Mass Variation)
用户询问是否输出 `mass`。
**决策**: `params.mass` **不应** 作为 Mamba 的输入特征。
- 理由：真实运行中 `mass` 是未知的（或需要估计的）。Mamba 应通过 `I_meas` (电流) 和 `accel_x` 的关系来隐式推断负载变化。
- 只有在以此为 *监督学习目标* 时才作为 Label 输出。

## 5. 结论与下一步计划

### 5.1 需要修改的代码 (P0级 - 必需)
1. **修改 `output_eq_ref.m`**:
    - **修复 `gyro_y_meas`**: 改为 `d(theta_ground)/dt + noise`，反映坡度变化。
    - **新增 `slip_ratio_lf`, `slip_ratio_rr`**:  `(omega*r - v) / v`，作为 "Slipping" 状态的强特征。
    - **新增 `accel_y_meas`**: 基于 `(Fy_f + Fy_r)/m` 的横向加速度读数。

### 5.2 暂不修改但需记录的 (P1级)
- `cornering_stiffness`: 维持 300，但标记为非物理。
- `state_eq` 中的人工阻尼: 维持现状。

### 5.3 Mamba 训练数据生成
- 无需修改 GRU 的生成脚本，只需确保新录制的数据集包含上述 P0 新增的通道。

## 6. 修改建议清单

### 6.1 `output_eq_ref.m` 修改细节

1.  **增加纵向滑移率输出**
    *   计算公式：`slip_ratio = (omega_wheel * r - v_x) / max(abs(v_x), 0.1)`
    *   目的：提供 slipping 程度的直接量测（模拟转速差）。

2.  **增加横向加速度输出**
    *   计算公式：`accel_y_meas = (Fy_f + Fy_r) / m + noise`
    *   目的：提供 turning 和 sliding 的动力学特征。

3.  **计算真实的 `gyro_y`**
    *   由于 `output_eq` 是无状态的，需要传入 `theta_ground` 的导数，或者在函数内部用 persistent 变量做一个差分近似。
    *   建议：`gyro_y = (theta_ground - theta_prev) / Ts + noise` (使用 persistent)。

4.  **`gyro_y` 实现补充**
    *   由于 `output_eq_ref.m` 是函数，需要使用 `persistent` 变量保存上一时刻的 `theta_ground`。代码示例如下：
    ```matlab
    persistent theta_ground_prev;
    if isempty(theta_ground_prev)
        theta_ground_prev = theta_ground;
    end
    gyro_y_clean = (theta_ground - theta_ground_prev) / Ts;
    theta_ground_prev = theta_ground; % Update
    gyro_y_meas = gyro_y_clean + noise;
    ```

### 6.2 移除建议
*   不要直接输出 `params.mass` 到 input features 中。

### 6.3 建议输出列表 (Revised Output Vector)

| 索引 | 变量名 | 状态 | 说明 |
| :--- | :--- | :--- | :--- |
| ... | 原有 1-31 | 保留 | ... |
| 32 | **`slip_ratio_lf`** | **[NEW]** | P0-1: 左前轮纵向滑移率 (带限幅) |
| 33 | **`slip_ratio_rr`** | **[NEW]** | P0-1: 右后轮纵向滑移率 (带限幅) |
| 34 | **`accel_y_meas`** | **[NEW]** | P0-3: 横向加速度 (基于侧向力) |

#### 实现细节备忘:
*   **滑移率限幅**: `slip_ratio = sat((w*r - v)/v, -1.0, 1.0)`，防止低速分母为零时的数值爆炸。
*   **横向加速度**: `ay = (Fy_front + Fy_rear)/mass + noise`。

### 6.4 其他一致性建议
1.  **低速阈值统一**: 检查 `state_eq` (0.05) 和 `output_eq` (1e-3) 的低速阈值，建议统一使用 `params.low_speed_threshold`。

## 8. 基于新路径的 AGV 模型参数适配分析

为确保 AGV 能顺利跑完上述 120s 工业路径，需检查并调整以下模型参数：

### 8.1 驱动力限幅 (Force Limits) - [建议直接修改]
*   **分析**: 现有驱动力限幅来自 MPC 输入约束（`opts.umax/umin`），默认 ±300N。在 10° 坡度下，仅重力分量就达 $341N$。
*   **如果不修改**: 在 10° 上坡时，电机输出饱和也无法克服重力，导致车辆失速 (Stall) 下滑，测试 F3 (坡度切换) 将直接失败。
*   **修改意见**: **必须修改**。将 MPC 约束提升至 **±600N**（见 [src/mpc/mpc_setup_single_interp.m](src/mpc/mpc_setup_single_interp.m#L86-L110)）。

### 8.2 侧偏刚度 (Cornering Stiffness) - [建议观察测试]
*   **分析**: 现有值 300 N/rad 极低。
*   **观察指标**: 运行 Closed-loop Test (场景 `straight_left_turn`)。
    *   如果 `e_y > 0.3m` (跟踪误差过大) 或出现振荡。
    *   如果 `slip_angle (beta)` 超过 5°。
*   **修改意见**: 若出现上述现象，说明轮胎“太软”无法提供足够转向力，建议将 `front/rear_cornering_stiffness` **逐步提升至 3000 N/rad**。

### 8.3 摩擦系数 (Friction Coefficient) - [建议观察测试]
*   **分析**: `mu = 0.8`。
*   **观察指标**: 运行 Open-loop Test (场景 F2 平地转弯) 或 Closed-loop 复合工况。
    *   如果 `slip_ratio_lf/rr` 接近或达到 1.0 (饱和打滑)。
    *   如果车辆实际转弯半径远大于理论半径 (推头)。
*   **修改意见**: 若出现打滑，说明地面附着力不足，建议将 `mu` **提升至 0.8**。

### 8.4 控制器前视 (MPC Horizon) - [建议观察测试]
*   **分析**: 当前参考路径转弯段与过渡长度（4s、1.2s）决定需要更长的预测时域。
*   **观察指标**: 在 S 弯工况下。
    *   如果 AGV 在入弯处总是“反应迟钝”，切角过大。
*   **修改意见**: 建议统一为 **$N_p \times T_s \approx 1.5s$**（即 `Np=150`，`Ts=0.01s`），兼顾过渡前瞻与计算量。

### 8.5 初始状态 (Initial Conditions) - [建议直接修改]
*   **分析**: 路径需从静止启动。
*   **修改意见**: 将 `parameters.m` 中 `initial_velocity` 设为 **0**（见 [src/core/parameters.m](src/core/parameters.m#L80-L94)）。

## 9. 线性化网格参数适配分析 (Linearization Grid)

为支持新路径的大转弯 (约 0.37 rad/s) 和鲁棒性坡度 (10°/0.175 rad) 工况，需检查 LPV 模型的网格覆盖范围。

### 9.1 角速度网格 (W_grid) - [必须修改]
*   **分析脚本**: `src/lpv/test_lpvmpc_workflow.m`
*   **现状**: `W_grid = linspace(-0.2, 0.2, 5)` rad/s。
*   **问题**: 新路径在转弯区的最大角速度约为 $\omega \approx 0.37$ rad/s (84°转弯/4s)。
*   **后果**: 
    1.  实际运行点 $\omega=0.37$ 超出网格边界 $0.2$。
    2.  需扩展网格以包含该工况。
*   **建议**: 修改 `test_lpvmpc_workflow.m`，将 `W_grid` 扩展为 `linspace(-0.45, 0.45, 7)`。范围适中，既覆盖峰值又保证插值精度。

### 9.2 坡度网格 (T_grid) - [保持现状]
*   **现状**: `T_grid = linspace(-0.2, 0.2, 5)` rad (约 $\pm 11.46^\circ$)。
*   **结论**: 已覆盖 $\pm 10^\circ$ (0.175 rad) 的极限工况，无需修改。建议保持该范围。

### 9.3 速度网格 (V_grid) - [建议扩展]
*   **现状**: `V_grid = linspace(0.8, 1.2, 5)` m/s。 
*   **问题**: 新路径的速度范围为 **0 ~ 1.0 m/s** (含启动区)。现有网格下限 0.8 m/s 无法覆盖 0-0.8 m/s 的启动阶段。
*   **建议**: 将 `V_grid` 扩展为 `linspace(0.1, 1.2, 5)`，以改善低速段的模型匹配精度（避免 v=0 的奇点，从 0.1 开始）。

### 9.4 动作项
*   **重新生成模型**: 修改网格范围后，**必须运行** `test_lpvmpc_workflow.m` 以重新生成并保存 `lin_agv_db.mat`。仅修改文本是不够的。

## 10. 自适应 MPC 参数适配分析 (Adaptive MPC)

针对新路径包含的 **S 弯 ($\omega \approx 0.37$ rad/s, R≈2.7–3.6m)** 和 **鲁棒性坡度 (10°)**，现有的自适应策略参数需要重新校准。

### 10.1 预测时域 (Prediction Horizon)
*   **文件**: `src/mpc/mpc_setup_single_interp.m` (或 `test_lpvmpc_workflow.m`)
*   **分析**: 默认 `opts.Np` 通常设为 30-50 步。对于 1.2s 的过渡过程，建议 $T_p \approx 1.5 \sim 2.0s$。
*   **修改意见**: 若 `Ts=0.01s`，建议设定 **`Np = 150`**。若计算量过大，可适当降低，但不能低于 100。

### 10.2 方案 B (角速度自适应) - [需调整增益]
*   **文件**: `src/mpc/mpc_setup_single_interp.m` (maps配置区)
*   **现状**: 
    - `omega_threshold = 0.15` rad/s (触发阈值)
    - `q_y_gain_max = 1.8` (最大增益)
*   **分析**: 新转弯工况 $\omega \approx 0.37$ rad/s，超过触发阈值。
*   **修改意见**:
    1.  **保持阈值**: `0.15` 合理，能灵敏触发。
    2.  **提升增益**: 建议将 `q_y_gain_max` 提升至 **2.5**，抑制 S 弯中的横向误差 (从 1.8 适度提升即可)。

### 10.3 方案 C (坡度自适应) - [需大幅增强]
*   **文件**: `src/mpc/mpc_update_from_rho.m`。
*   **现状**: 
    - `theta_threshold = 0.035` rad (2°)
    - `theta_transition_width = 0.017` rad (1°)
    - `q_v_gain_max = 1.5`
    - `R_F_gain_max_uphill = 1.2`
*   **分析**: 面对 **10° (0.175 rad)** 的极限坡度，重力干扰极大。现有的 1.2 倍输入权重 R 调整可能太保守，且 q_v 增益不足。
*   **修改意见**:
    1.  **阈值**: 保持 2° 即可。
    2.  **q_v 增益**: 提升至 **3.0**，确保坡道不掉速。
    3.  **R(1) 增益**: **这是关键**。上坡时我们需要电机“敢于输出”，不应过度惩罚输入。建议 `R_F_gain_max_uphill` 设为 **0.5 (减小权重)** 而不是增大权重，或者保持 1.0。
    *   *修正*: 通常上坡需要更大的控制力，如果 R 是惩罚项，减小 R 会允许更大的输入。**请确认 R 的定义**。若 R 是惩罚项，上坡应**减小 R** (允许猛烈出力) 或 **保持不变但大幅增加 Q_v**。现有逻辑 "上坡增大 R" 可能是为了防止打滑？
    *   **最终建议**: 上坡时主要靠提升 **q_v (速度误差权重)** 来克服重力，建议 `q_v_gain_max = 5.0`。R 保持现状防止打滑。

### 10.4 软约束 (Soft Constraints)
*   **分析**: 默认位置软约束权重 `1e4`。
*   **修改意见**: 在 10° 坡道+转向的极端工况下，硬约束极易不可行。建议确保位置 `e_y` 和航向 `e_psi` 始终为 **软约束**，且适当降低权重 (e.g., `1e3`) 以防止求解器在不可行解处卡死（Infeasible）。

## 11. 测试脚本增强建议

为全面验证上述参数调整的效果，建议对测试脚本进行以下增强：

1.  **Open-loop (test_agv_open_loop.m)**:
    *   新增 **[G] 极限爬坡测试**: 设定坡度 10°，给定恒定驱动力 (e.g., 500N)，验证是否能从静止启动并达到稳态速度。验证 `max_force=600N` 的必要性。

2.  **Closed-loop (test_simulink_closed_loop.m)**:
    *   **集成 Industrial Path**: 必须新增一个测试场景，加载并运行完整的 120s `industrial` 路径，验证全流程通过性。
    *   **网格覆盖率检查 (Grid Coverage)**: 绘制实际运行的 $\rho=[v, \omega, \theta]$ 轨迹与线性化网格边界的对比图。验证 $\omega$ 是否超出 $\pm 0.45$ rad/s，以及 $v$ 是否落入 $0.1 \sim 1.2$ m/s 范围。
    *   **控制饱和度统计**: 统计 `F_cmd` 达到限幅 (600N) 的时间占比，评估驱动系统的余量。
    *   **动力学极限监控**: 输出 **侧偏角 (Beta)** 和 **滑移率 (Slip Ratio)** 的峰值，确保在 S 弯和 10° 坡道上车辆未失稳 (建议 Beta < 5°, Slip < 1.0)。
